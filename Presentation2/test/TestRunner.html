Using HeDM on the Ellucian Hub

While there are many conceptual similarities between our products, there is a lot of variation in how things are named and organized. What is required by Banner may not be required by Colleague - and may not even be a concept at all in Recruiter or Elevate. The following are some best practices for using HeDM in these scenarios.

Interaction between systems is done through the Ellucian Hub, which uses a Publisher/Subscriber model.

  • Publishers:
    • Are authoritative for specific HEDM entities.
    • Persist or provide defaults for the full HEDM model, all properties.
    • Have exclusive permission to create, update, and delete.
  • Subscribers:
    • If syncing, subscribers are keeping a copy of published data.
    • If consuming, subscribers are reading published data.
    • No subscribers are authoritative, they only have permission to read.

When requesting a list of matching entities via the API (e.g., /persons?[query-string], the following properties are required to be allowed in query strings. The format of the query-string must follow the API Strategy Guide, and use the "unstructured", name=value form.

The JSON Schema for each entity listed here must provide a JSON Hyper-Schema link named "filter" that documents the allowed submission query parameters.

The response to all filter queries is an array of objects containing a "guid" property of all matching entities, or an empty array if there are no matching results.

  • Person
    • Name (Support filtering on all name fields)
    • Roles
    • Credentials - Banner ID, Banner Sourced ID, Banner User Name, Banner UDC ID
    • EmailType and EmailAddress
    • Filter based on Population Selection
    • Course
      • Subject and Number
      • academicLevels
      • owningOrganizations (College, Department)
      • Title
      • instructionalMethods
      • Credit
      • effectiveStartDate
      • effectiveEndDate
    • Section
      • Title
      • startDate
      • End Date
      • Code
      • Number
      • instructionalPlatform
      • academicPeriod
      • course (filter all sections for a given course guid)
      • site
      • academicLevels
      • status
      • owningOrganizations
      • integrationpartner (ILP requirement but not yet added to v2 HeDM)
    • Instructional-events
      • Section (filter all instructional-events for a section)
      • startDate
      • End Date
      • Location.Room
      • instructorRoster.instructor
      • Paging

Regarding partial updates in HEDM APIs:

Systems must support partial updates regarding optional HEDM defined fields in PUT operations at this time. This means, when processing a PUT operation, only those fields present in the request JSON content body should be updated(or removed) from the resource. Systems are not required to implement the HTTP PATCH verb at this time. Note that this caveat is noted in the Ellucian restful API strategy:

"Note: 'PUT' may, in the future, be replaced with 'PATCH' as 'PUT' is actually intended to replace the resource based upon the new content whereby 'PATCH' is intended to 'update' the existing resource with new content. REST APIs should use 'PUT' with 'PATCH' semantics until PATCH is more broadly supported."

To further clarify, this means that HEDM API behavior on PUT should be to overlay the fields provided in the request over the existing resource only changing fields which were actually provided in the original web request body. We handle any property present in the content body and use NULL(or empty string for string literals) to note the removal or deletion of a value/associated object. This behavior is still compatible with full representations where all HEDM fields are provided, although does not wipe out fields unmentioned in the content body like HTTP PUT specification dictates.

We will revisit this standard once HTTP PATCH becomes more widely supported in the REST API industry as well as within commonly used client and server framework SDK versions.

Things needing consideration when revisiting this standard from an HEDM prospective:

Change events by BEP/EDX must identify between updates and creation of a new resource. (An insert of an address is not the creation of a new person. Also the insert of a new name record and update invalidating an old name record for audit history is not the creation of a new person)

Update content bodies will need to be generated by the ERP adapter off of the event as no restful-api in the source system will build this differential JSON formatted message.

Each implementer of the HEDM models in a restful API should provide their own data dictionary for clients.

All HEDM models will be centrally documented on the XE Registry where Banner and Colleague API resides.

HEDM defines the schema which describes the type and whether required for each field in the HEDM model. Also has example responses for each endpoint.

Filterable fields are defined in the links section of the response schema.

Special query requests sent via POST through QAPI need their own schema.

In order to differentiate different special requests on the same HEDM resource, the Content-Type must be provided which explicitly defines the request type sent and which schema the request body should be validated against. This allows for each special request to have it's own schema with it's own required and optional properties defined.

For persons when duplicate checking the schema is the same as the normal person schema. However, when requesting a saved person list (person-filter) a required property named person-filter is required of type string and format guid.

This allows us to have multiple different special requests that return standard HEDM resources from the standard HEDM endpoint for that resource.

Special request suggested naming:

Media Type Schema file Resource being called
application/vnd.hedtech.integration.duplicate-check.v1+json persons_duplicate-check.json /persons
application/vnd.hedtech.integration.person-filter.v1+json persons_person-filter.json /persons
application/vnd.hedtech.integration.room-availability.v1+json rooms_room-availability.json /rooms

Responses also may have extended or minimal representations.
